Sección 1 — Conceptos

01

Conceptos de IA vistos hoy

Las piezas del rompecabezas: tres formas de usar Claude, qué es una skill, qué es MCP, y por qué un "Project" de chat no es lo mismo que un agente real.

↓ para profundizar
01 — Tres formas de usar Claude

Un Claude, tres maneras de trabajar

💬 Chat

Conversación simple. Vos hablás, Claude responde. Nada pasa hasta que escribís.

Para: pensar, decidir, redactar algo corto.

🗂️ Cowork

Le das una carpeta y una tarea. Trabaja solo (sub-agentes en paralelo) y entrega el resultado.

Para: reportes, reconciliar planillas, tareas de oficina.

💻 Code

Escribe, corrige y corre código real. Acá viven los agentes/skills que armamos hoy.

Para: programar, automatizar, construir herramientas.

Clave: los tres comparten el mismo modelo — cambia el nivel de autonomía y de acceso a herramientas.
01 — Skill = receta guardada

Una skill es un instructivo que Claude sigue siempre igual

Un archivo SKILL.md guardado en una carpeta. Escribís su comando (ej. /reveal-minilesson) o una frase parecida, y Claude sigue esos pasos exactos, cada vez.

Ejemplo de hoy: creamos /explica-esto-2p — cada vez que la corrés, genera 2 láminas con el mismo estilo visual, sin que expliques de nuevo el formato.
01 — MCP = enchufe universal

MCP conecta Claude con tus apps reales

Antes: una integración a medida por cada herramienta (Drive, Notion, Figma...). Con MCP, todas hablan el mismo protocolo — un mismo "enchufe" en vez de un cable distinto para cada una.

Vos autorizás qué puede leer, crear o modificar en cada herramienta conectada.
01 — Project ≠ agente real

Un Project (chat) no actúa solo

Un Project guarda instrucciones y documentos — memoria persistente. Pero sigue esperando que vos le escribas cada vez, como cualquier chat.

Un agente real (subagente de Claude Code, o Cowork) recibe una tarea grande, usa herramientas, decide los pasos, y trabaja sin que estés mirando.

Diferencia clave: Project espera tu mensaje cada vez; el agente actúa solo.
Sección 2 — Agentes reales

02

Cómo armamos agentes reales

De la teoría a la práctica: subagentes de Claude Code, con archivos, reglas propias, y acceso controlado a herramientas.

↓ para profundizar
02 — Estructura de un subagente

Un archivo Markdown con reglas

// .claude/agents/nombre-del-agente.md
---
name: nombre-del-agente
description: cuándo usarlo (condición de disparo)
tools: Read, Bash, Glob
---
(acá abajo va el system prompt completo del agente)

description es lo más importante — Claude lo lee para decidir cuándo delegar. tools es tu control de seguridad: lo que no le des, no lo puede hacer.

02 — Regla de oro

Releer siempre, no inventar nunca

1
Confirmar que los archivos de entrada existen
Glob + verificación antes de calcular
2
No confiar en una corrida anterior
cada ejecución relee todo desde disco
3
Si falta un dato, avisar — nunca inventarlo
crítico cuando el resultado mueve decisiones de compra o producción

03 — La lógica

Punto de reorden + nivel objetivo

Punto de reorden = consumo durante (tiempo de entrega + 7 días de colchón). Si el stock cae por debajo, hay que comprar YA.

Nivel objetivo = consumo durante (30 días de ciclo + lead time + colchón). La cantidad recomendada es lo que falta para llegar ahí — redondeado a cajas completas y nunca menor al mínimo del proveedor.

Sección 4 — Capacidad JIS

04

Análisis de capacidad: 120 → 300 autos/turno

Alimentación secuenciada a línea — espumas, fundas y cables — sin buffer ni kanban. ¿Aguanta el proceso actual al doble y medio de volumen?

↓ para profundizar
04 — El proceso y su restricción

Cinco pasos, un ciclo, cero stock

Picking → Secuenciación → Traslado a línea (troley) → Descarga → Retorno → se repite.

Restricción dura: se respeta siempre la secuencia exacta de modelos que pide la línea. Nada de buffer, nada de kanban.
04 — El modelo (validado con tus números)

De autos/hora a runners necesarios

autos/hora = autos por turno / horas netas
piezas/hora = autos/hora × piezas por auto
viajes/hora = piezas/hora / capacidad por viaje
min. requeridos/hora = viajes/hora × tiempo por viaje
runners necesarios = min. requeridos / 60
Validado: con tus datos reales de fundas (8 fundas/auto, 30/viaje, 10 min/viaje) el modelo reprodujo exacto tu "no go — hacer dos equipos".
04 — Resultado por familia

¿Quién aguanta 300 y quién no?

FamiliaActual (120)Objetivo (300)
Fundas (real)OK — 1 runnerNO GO — necesita 2
Espuma (ejemplo)OK — 1 runnerNO GO — necesita 2

Total: 3 runners hoy → 5 runners necesarios a 300/turno, con el diseño actual sin rediseñar nada.

04 — Palancas recomendadas

Cerrar la brecha sin buffer ni kanban

1
Paralelizar por familia (espuma / funda / cable)
2
Reducir traslado y retorno — acercar secuenciación al punto de uso
3
Tren logístico (mizusumashi) con mayor capacidad, no más stock
4
Poka-yoke de secuencia — escaneo en picking + inducción a línea
5
Automatizar el retorno (AGV/AMR)
6
Sumar headcount — último recurso, no el primero
