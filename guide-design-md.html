Section 1 — What Is It

01

What Is It?

DESIGN.md solves the problem of AI agents inventing brand colors. One file, one source of truth, any agent.

↓ to dive in
What Is It · The Problem

The problem it solves

❌ Without DESIGN.md

  • AI invents colors every session
  • No consistent spacing or typography
  • Every "create a button" prompt yields different results
  • You re-explain brand tokens in every prompt

✅ With DESIGN.md

  • AI reads your exact hex codes, font stacks, spacing scale
  • Consistent output across every agent, every session
  • Token validation catches broken refs and contrast failures
  • One-command export to Tailwind or CSS
What Is It · File Structure

File structure — two layers

A single DESIGN.md file in your project root. Agents pick it up automatically.

--- ← YAML front matter (machine-readable tokens)
version: "alpha"
name: "My Project"
colors:
  primary: "#f5a623"
typography:
  body:
    fontFamily: "Inter, sans-serif"
components:
  button:
    backgroundColor: "{colors.primary}" ← token reference
---

## Overview ← Markdown body (human-readable rationale)
Primary amber is for CTAs only. Never use it decoratively.
Token references: Use {section.token-name} syntax to reference other tokens. The linter validates all references — broken refs are caught at lint time.

Installation

Install the CLI globally via npm. One command, then you have lint, diff, export, and spec.

↓ install → verify → Windows note
Installation · npm

Install globally

# Install globally (recommended)
npm install -g @google/design.md

# Or run one-off without installing
npx @google/design.md lint DESIGN.md
1
Run the install command above
npm install -g @google/design.md → adds 92 packages
2
Verify the CLI is available
designmd --version
3
Create your first DESIGN.md and lint it
designmd lint DESIGN.md
npm fund message: You'll see "74 packages looking for funding" after install. This is informational only — maintainers requesting optional support. Safe to ignore.
Installation · Windows

Windows — use the designmd alias

On Windows PowerShell, .md file extension conflicts with file association. Use designmd instead of @google/design.md for all commands.

# ❌ This may fail on Windows PowerShell:
npx @google/design.md lint DESIGN.md

# ✅ Use the alias instead:
designmd lint DESIGN.md
designmd diff DESIGN.md DESIGN-v2.md
designmd export --format css-tailwind DESIGN.md
designmd spec --rules
All examples in this guide use the designmd alias. On Mac/Linux, both forms work identically.

Token Schema

The YAML front matter supports 5 token types: colors, typography, spacing, rounded, components.

↓ each token type
Token Schema · Colors & Dimensions

Colors, Spacing & Rounded

colors:
  primary: "#f5a623" ← hex
  surface: "oklch(98% 0.01 240)" ← oklch
  overlay: "rgba(0,0,0,0.5)" ← rgba

spacing:
  sm: "8px" ← number + unit (px, rem, em)
  md: "16px"
  lg: "32px"

rounded:
  sm: "4px"
  md: "8px"
  full: "9999px"

Typography tokens

    fontFeature: "normal" ← optional

Token Schema · Components

Component tokens — valid sub-tokens only

⚠ Only these 8 sub-token keys are accepted. All others produce warnings.

Sub-tokenAcceptsExample
backgroundColorColor value or token ref"{colors.primary}"
textColorColor value or token ref"{colors.text}"
typographyTypography token ref"{typography.body}"
roundedDimension value or token ref"{rounded.md}"
paddingDimension value or token ref"{spacing.md}"
sizeDimension value"48px"
heightDimension value"40px"
widthDimension value"100%"
Common mistake: Using background, color, border, or radius instead of their valid equivalents. The linter flags these as warnings.
Section 4 — CLI Commands

04

CLI Commands

Four commands: lint, diff, export, spec.

↓ command by command
CLI · lint

lint — validate your file

designmd lint DESIGN.md
designmd lint --format json DESIGN.md ← machine-readable output

9 lint rules

RuleSeverityChecks
broken-referrorToken references like {colors.x} that don't resolve
missing-primarywarningNo primary color defined — agents may auto-generate one
contrast-ratiowarningColor pairs that fail WCAG AA (4.5:1 text, 3:1 UI)
orphaned-tokenswarningColors defined but never referenced by any component
missing-typographywarningNo typography tokens defined at all
section-orderwarningMarkdown sections out of canonical order
unknown-keywarningUnrecognized keys — likely typos in sub-token names
missing-sectionsinfoOptional Markdown sections absent
token-summaryinfoCount of colors, typography scales, components defined
CLI · diff & spec

diff — compare versions

designmd diff DESIGN.md DESIGN-v2.md
↳ token-level diff: what changed between versions
↳ catches color regressions before they reach production

spec — dump the format spec

designmd spec # full spec output
designmd spec --rules # lint rules only — paste into agent system prompt
Use spec --rules when setting up a new AI agent. Paste the output into your system prompt so the agent understands the DESIGN.md format.
CLI · export

export — convert tokens to target format

CommandOutput
export --format css-tailwind Tailwind v4 CSS @theme block with CSS custom properties
export --format json-tailwind Tailwind v3 theme.extend JSON object
export --format dtcg W3C Design Token Community Group standard tokens.json
# Export Tailwind v4 CSS → redirect to file
designmd export --format css-tailwind DESIGN.md > src/tokens.css

# Export Tailwind v3 JSON
designmd export --format json-tailwind DESIGN.md > tailwind.tokens.json

# Export W3C DTCG
designmd export --format dtcg DESIGN.md > tokens.json

05

Workflows

Create · Validate · Export · Use with AI agents · Version control

↓ workflow by workflow
Workflows · Create from Scratch

Workflow 1 — Create DESIGN.md for a project

1
Identify your existing design tokens
Check existing CSS variables, Figma Variables, Tailwind config, or HTML files
2
Create DESIGN.md in your project root
Add version, name, colors, typography, spacing, rounded, components in YAML front matter
3
Add Markdown rationale sections below the front matter
## Overview, ## Colors, ## Components — document usage rules
4
Lint and fix
designmd lint DESIGN.md → fix all errors → target 0 errors, minimal warnings
5
Commit to repo root
AI agents discover DESIGN.md automatically when it's at project root
Workflows · Validate & Export

Workflow 2 — Validate and export tokens

1
Run lint to catch all issues
designmd lint DESIGN.md → review errors first, then warnings
2
Fix errors (broken refs, unknown keys)
Check token reference paths — {colors.x} must match a key in colors:
3
Re-lint until 0 errors
Warnings on orphaned palette tokens are acceptable — they're palette reference colors
4
Export to your target format
designmd export --format css-tailwind DESIGN.md > src/tokens.css
5
Import exported file in your build system
@import './tokens.css' in your main CSS / extend tailwind.config.js
Workflows · Version Control

Workflow 3 — Version control design changes

1
Save the current DESIGN.md as a versioned copy before updating
cp DESIGN.md DESIGN-v1.md
2
Make your token changes in DESIGN.md
Update colors, spacing scale, component tokens as needed
3
Run diff to see exactly what changed
designmd diff DESIGN-v1.md DESIGN.md
4
Review diff in PR — catch regressions before merging
Changed primary color? Removed a token still used in components? diff catches it.
Section 6 — Limits & Tips

06

Limits & Tips

Alpha-stage caveats and patterns that produce the best results.

↓ limits → tips
Limits & Tips · Limits

Known limits alpha

LimitDetail
Component sub-tokens restricted Only 8 sub-tokens accepted: backgroundColor, textColor, typography, rounded, padding, size, height, width. All other CSS properties produce unknown-key warnings.
Windows alias required PowerShell conflicts with .md extension. Always use designmd alias, never @google/design.md directly in PowerShell.
Alpha spec — breaking changes possible Token schema and lint rules may change between releases. Pin your version in package.json for production use.
Orphaned color warnings expected Palette colors not referenced in any component trigger warnings. Acceptable for colors used only in Markdown rationale (status colors, data-vis palette, etc.).
No hover / state variants Components can't express hover, focus, disabled, or active states in the token schema. Document these in the Markdown ## Components section instead.
Limits & Tips · Tips

Tips for best results

  • Always define primary in colors — agents use it as the default focal color
  • Use token references in components ({colors.primary}) not raw hex — keeps tokens consistent and refactor-safe
  • Run lint in CI — add designmd lint DESIGN.md as a pre-commit hook or CI step
  • Use diff in PRs — any design token change should include a designmd diff output in the PR description
  • Extract from existing files first — read your CSS variables or Figma file before writing tokens manually
  • Pair with Figma MCP — use get_variable_defs to pull live Figma Variables and auto-populate DESIGN.md
  • designmd spec --rules → paste into agent system prompt for any LLM that writes DESIGN.md files
